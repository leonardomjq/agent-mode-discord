<pitfalls>
- **Pitfall 4 (RESEARCH):** Linux fs.watch does NOT support `{ recursive: true }`. Always check `process.platform === "linux"` and use polling-stat instead. Failure mode if you skip this: detector silently does nothing on Linux.
- **PRD §FR-1.8 ban:** NEVER read JSONL content. Only `fs.statSync(...).mtimeMs`. The undocumented Anthropic format will break and we'd be coupled to it. The reviewer will reject any `readFileSync` on a .jsonl path.
- **Pitfall: directory may not exist at startup:** Users without Claude installed won't have ~/.claude/projects/. Don't throw, don't dispatch, just poll for the directory to appear (5 s setInterval checking existsSync). When it appears, transition into watch mode.
- **macOS double-fire:** fs.watch on macOS fires twice for one rename. The 100 ms debounce is mandatory. Without it, the rescan runs twice per change which doubles dispatch traffic.
- **Pitfall 6 (RESEARCH commandLine value):** N/A here — sessionFiles doesn't touch shell integration.
- **stat race:** A file may be deleted between `readdirSync` and `statSync`. Wrap statSync in try/catch and just skip the entry on ENOENT (the next rescan will catch it if recreated).
- **Two parallel claude sessions DET-04:** sessionFiles aggregates to a single `agent-started/agent-ended` because it can't distinguish per-terminal at this layer. The orchestrator (Plan 03-04) handles per-terminal aggregation by combining tiers; sessionFiles just signals "at least one Claude process is alive somewhere on this machine".
- **Cleanup race on dispose:** clear all timers BEFORE calling watcher.close(). Otherwise a timer fires post-dispose and dispatches into a stale callback.
</pitfalls>
