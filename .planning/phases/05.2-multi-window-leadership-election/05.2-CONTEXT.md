# Phase 05.2: Multi-window leadership election — Context

**Gathered:** 2026-04-16
**Status:** Ready for planning
**Mode:** `--auto` single-pass (Claude auto-selected; user pre-approved the architectural call: "show the first one open, simple")

<domain>
## Phase Boundary

Replace the v0.1.0 multi-window behavior — currently "every VS Code window opens its own Discord RPC connection with a different pid; Discord's desktop client picks one to display, typically the most recently updated" — with a single-leader design: the first VS Code window to start owns Discord presence; subsequent windows detect the leader and stay quiet.

**In scope:**
- New module: `src/state/leadership.ts` — leader-acquire, keep-alive heartbeat, release on dispose
- Wire-in: `src/extension.ts` activate() acquires leadership BEFORE starting `ConnectionManager`; followers skip RPC entirely
- Lockfile path: `~/.claude/agent-mode-discord.leader.lock` (mirrors companion lockfile location, same `~/.claude/` dir)
- Heartbeat: 30s mtime touch on the lockfile by the leader
- Stale-leader takeover: if a follower observes lockfile mtime older than 90s (3× heartbeat), it steals leadership (handles crashed-leader case)
- Tests: unit tests for `leadership.ts` covering acquire / contention / stale-takeover / release
- Update: `docs/MULTI-WINDOW.md` — replace "depends on Discord's merging" content with "first window wins; subsequent are quiet" + the test matrix expectations

**Out of scope:**
- Per-window per-workspace differentiation (e.g. "show the focused window's workspace") — over-engineering for v0.1.0
- Cross-process leader election protocols (Raft, etc.) — lockfile + mtime is sufficient
- Notification UI when a window becomes a follower (silent posture per D-18)
- Release workflow integration (Phase 6 owns)
- Telemetry on leadership transitions (SEED-001 future)

</domain>

<decisions>
## Implementation Decisions

### Lockfile path and pattern
- **D-01:** Lockfile at `~/.claude/agent-mode-discord.leader.lock`. Mirrors the existing companion lockfile dir (`~/.claude/`) for consistency. Distinct filename so the companion detector (which watches `agent-mode-discord.lock`) cannot collide. The `.leader` segment makes intent obvious to anyone inspecting `ls ~/.claude/`.
- **D-02:** Acquire via `fs.promises.open(path, 'wx')` (atomic create-fail-if-exists). On `EEXIST`, the path was already taken by another window. No `try mkdir + write` race window — `wx` is atomic at the OS level on macOS / Linux / Windows.

### Heartbeat and staleness
- **D-03:** Leader touches the lockfile mtime every 30 seconds via `fs.promises.utimes`. Followers check mtime; staleness threshold is 90 seconds (3× heartbeat) — gives the leader 2 missed heartbeats before steal.
- **D-04:** Stale-takeover is greedy: the first follower to observe a stale lockfile attempts `unlink + open(wx)`. Race-safe because the `unlink` succeeds for at most one process; the others lose the `open(wx)` race and remain followers.

### Follower behavior
- **D-05:** Followers do NOT instantiate `ConnectionManager`. They register a poll (every 30s) to check whether the leader's lockfile is gone or stale; on takeover, they bootstrap the full `ConnectionManager` and become the new leader. This keeps follower extensions effectively idle (zero Discord IPC activity, zero CPU beyond a 30s setInterval).
- **D-06:** When a follower takes over, it does NOT replay any state from the previous leader (no shared state across processes). It starts fresh — current state is `IDLE` from `initialState()`. The detectors will re-fire when activity resumes.

### Release semantics
- **D-07:** On `deactivate()`, the leader unlinks the lockfile if `isLeader` is true. Failure is silent (D-18). If the unlink fails (permissions, race), the file becomes stale within 90s and another window steals it normally.
- **D-08:** If the process crashes (no `deactivate()` runs), the lockfile becomes stale within 90s. No PID liveness check — purely mtime-based. Rationale: a PID check would be platform-specific (`kill -0` on POSIX, `OpenProcess` on Windows) and the 90s window is acceptable for a presence extension that's not safety-critical.

### Module shape
- **D-09:** New file `src/state/leadership.ts` exports `createLeadership({ lockfilePath?, heartbeatMs?, stalenessMs?, fs?, now?, setInterval?, clearInterval? }) → { acquire(): Promise<boolean>, release(): Promise<void>, isLeader(): boolean, onTakeover(cb: () => void): void }`. Same dependency-injection pattern as `src/detectors/companion.ts` — keeps tests pure-core.
- **D-10:** `extension.ts` calls `leadership.acquire()` BEFORE `mgr.start()` and `activityBuilder.start()`. If `acquire()` returns `false`, register a poll for takeover via `leadership.onTakeover(...)` and return without starting RPC. If takeover happens later, lazily instantiate the same `mgr` + `activityBuilder` flow.

### Tests
- **D-11:** Tests in `test/state.leadership.test.ts` follow the established pure-core pattern (no `vscode` import). Cover: acquire-when-empty, acquire-when-held, stale-takeover, heartbeat-touches-mtime, release-unlinks, dispose-cleans-up. Inject `fs`, `now`, `setInterval`, `clearInterval` via constructor for deterministic time.

### Docs
- **D-12:** Rewrite `docs/MULTI-WINDOW.md` — drop the "expected behavior depends on Discord" framing, replace with "first window owns presence; others stay silent; if the leader exits or crashes, takeover happens within 90s". Update the test matrix to match (single-leader expectations).

### Observability and edge cases (Claude's Discretion)
- **D-13:** No verbose logging by default. If `agentMode.debug.verbose` is true, log "leadership acquired" / "leadership released" / "leadership taken over from stale leader" via `outputChannel.log`. Avoids noise for normal users; useful for the inevitable "why isn't my Discord status updating" GitHub issue.

<specifics>
## Specific Ideas

- The user's framing was "if the user has multi-window, we just show the first one open, simple." → translates to: the FIRST window to acquire the lockfile is the leader, regardless of which window has focus. No focus-tracking complexity.
- Heartbeat 30s + staleness 90s = 3× heartbeat. Standard "give it two missed beats" robustness pattern.
- This pattern intentionally accepts a 0-90s "no Discord activity" window after a leader crashes. For a presence extension, that's fine. For a payment system, it wouldn't be.
- The `~/.claude/` directory is already user-writable (companion detector relies on this). No new permission concerns.

</specifics>

## Deferred Ideas

- **Cross-window message bus** (post-v1.0) — if we ever want to show "5 windows open, currently working on X across N of them", we'd need IPC between windows. Out of scope; lockfile is sufficient for the stated UX.
- **PID liveness check** (post-v1.0) — would shrink takeover window from 90s to ~instant on graceful exit, but adds platform-specific code. Defer until users complain.
- **Per-workspace presence differentiation** (post-v1.0) — show different presence for different workspaces by hashing workspace path into the lockfile name. Out of scope: the user explicitly chose "show the first one, simple."
- **Telemetry on leadership transitions** — folded into SEED-001 when extension-telemetry lands
