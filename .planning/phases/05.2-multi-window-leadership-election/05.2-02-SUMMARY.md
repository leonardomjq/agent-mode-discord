# Phase 05.2 Plan 02: Wire Leadership Election into extension.ts — Summary

Wire `createLeadership` (delivered by 05.2-01) into `src/extension.ts` so the first VS Code window acquires Discord presence leadership while subsequent windows remain silent followers — with a lazy-bootstrap takeover path if the leader crashes.

## What Was Built

### Leader / follower split in createDriver()

`createDriver()` now:

1. Calls `readConfig()` and `setVerboseCache()` synchronously (must precede acquire so the log sink works).
2. Instantiates `createLeadership({ log: verboseLog })` (D-13).
3. Fires `void leadership.acquire().then(isLeader => ...)` — keeping activate() fully synchronous (D-10).
4. **Leader path** (`isLeader === true`): logs "acquired" and calls `bootstrapAsLeader()` which delegates to `bootstrapLeader()` in `leaderDriver.ts`.
5. **Follower path** (`isLeader === false`): logs "held by another window", registers `leadership.onTakeover(() => bootstrapAsLeader())` — no RPC created (D-05).
6. `dispose()` always awaits `leadership.release()` first (safe no-op for followers), then optional-chains all disposables under a `bootstrapped` guard.

### bootstrapLeader() in src/leaderDriver.ts

The full driver setup (ConnectionManager, throttle, activityBuilder, detectors, configListener, signal handlers) lives in `bootstrapLeader(ctx, bootCfg, _context)`.

- Accepts a `DriverCtx` mutable reference — extension.ts reads/writes the same fields via `ctx.*`.
- `ctx.bootstrapped` is set to `true` on entry, making the function idempotent (T-05.2-05 guard).
- `ctx.state = initialState()` on entry — fresh state, no replay from previous leader (D-06).
- All disposable references (`mgr`, `activityBuilder`, `configListener`, `editorDisposable`, `gitDisposable`, `detectorsDisposable`) are assigned to `ctx.*` fields so `dispose()` in extension.ts can reach them.

### Guard variables

| Variable | Type | Purpose |
|---|---|---|
| `ctx.bootstrapped` | `boolean` | Idempotency — prevents double-bootstrap if acquire() and onTakeover() race |
| `ctx.shuttingDown` | `boolean` | Prevents zombie bootstraps after deactivate() (T-05.2-05) |
| `ctx.mgr` | `ConnectionManager \| undefined` | undefined for followers; set in bootstrapLeader() |
| `ctx.activityBuilder` | `{...} \| undefined` | undefined for followers |
| `ctx.configListener` / `ctx.*Disposable` | `vscode.Disposable \| undefined` | optional-chained in dispose() |
| `ctx.idleTimer` | `NodeJS.Timeout \| null` | cleared in dispose() regardless of leader status |
| `ctx.state` | `State` | local to leader epoch; reset via initialState() on each bootstrap |

### Decisions implemented

- **D-05** — Followers never call `createConnectionManager()`. Zero Discord IPC from follower windows.
- **D-06** — `ctx.state = initialState()` at bootstrap entry. No shared state with prior leader.
- **D-10** — `leadership.acquire()` is the first async operation inside createDriver; `mgr.start()` is the last call in bootstrapLeader, always after acquire resolves true.
- **D-13** — Every leadership log line uses `{ verboseOnly: true }`. Messages: "acquired — bootstrapping as leader", "held by another window — this window is a follower", "taken over from stale leader — bootstrapping".
