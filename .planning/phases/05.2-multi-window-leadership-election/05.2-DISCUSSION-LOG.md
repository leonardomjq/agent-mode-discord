## Lockfile location

| Option | Description | Selected |
|--------|-------------|----------|
| `~/.claude/agent-mode-discord.leader.lock` (Recommended) | Mirrors companion detector dir; distinct filename to avoid collision | ✓ |
| `~/.config/agent-mode-discord/leader.lock` (XDG-compliant) | Cross-platform XDG, but introduces a new dir scheme | |
| OS temp dir | Cleanest semantically (transient state) but `/tmp` doesn't survive logout on macOS — false stale-leader takeovers across sessions | |

**Auto-selected:** Option 1. Consistency with the companion lockfile location wins; users already have `~/.claude/` because of Claude Code itself.

---

## Acquisition mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| `fs.promises.open(path, 'wx')` (Recommended) | Atomic create-fail-if-exists at OS level; standard POSIX/Windows behavior | ✓ |
| `try { stat() }` then `writeFile` | Has TOCTOU race window | |
| Lock library (`proper-lockfile`) | New runtime dep; violates "one runtime dep" PROJECT.md decision | |

**Auto-selected:** Option 1. The single-line `fs.promises.open(path, 'wx')` does exactly what we need with zero deps.

---

## Heartbeat + staleness threshold

| Option | Description | Selected |
|--------|-------------|----------|
| 30s heartbeat / 90s staleness (Recommended) | 3× heartbeat — robust to one missed beat | ✓ |
| 10s heartbeat / 30s staleness | Faster takeover but more wake-ups; aggressive | |
| 60s heartbeat / 180s staleness | Lower wake-up cost; 3-minute lag on takeover too noticeable | |

**Auto-selected:** Option 1. 30s/90s balances takeover lag (acceptable for presence) and CPU wake-up cost.

---

## Follower poll cadence

| Option | Description | Selected |
|--------|-------------|----------|
| 30s setInterval matching heartbeat (Recommended) | One source of truth for "leadership rhythm" | ✓ |
| `fs.watchFile` on lockfile | Reactive; matches companion detector. Slightly higher complexity for a simple polling case | |
| Event-driven via cross-process signal | Out of scope — would need IPC layer | |

**Auto-selected:** Option 1. setInterval is the simplest correct answer; `fs.watchFile` is over-engineered for a 90s tolerance.

---

## Stale-takeover greedy vs coordinated

| Option | Description | Selected |
|--------|-------------|----------|
| Greedy: any follower can attempt takeover (Recommended) | Race-safe via `unlink + open(wx)` — one process wins per round | ✓ |
| Coordinated: lowest-pid takes over | Adds cross-process awareness; not necessary for correctness | |

**Auto-selected:** Option 1. The race-safety of `open(wx)` makes greedy correct without coordination.

---

## Follower idle behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Skip ConnectionManager entirely (Recommended) | Zero Discord IPC traffic; ~zero CPU — true follower | ✓ |
| Connect to Discord but never setActivity | Wastes a Discord IPC connection slot per window | |
| Connect and setActivity quietly (e.g. blank state) | Conflicts with leader's activity in Discord | |

**Auto-selected:** Option 1. A follower extension is a no-op extension — it just polls for takeover.

---

## State carryover on takeover

| Option | Description | Selected |
|--------|-------------|----------|
| Start fresh from initialState() (Recommended) | Simple; no shared state across processes | ✓ |
| Read leader's last state from lockfile contents | Adds serialization complexity for marginal UX gain | |
| Read VS Code workspace and start from CODING | Couples leadership to detector logic; breaks separation | |

**Auto-selected:** Option 1. Detectors will re-fire when activity resumes; clean restart is simpler than partial sync.

---

## Crash recovery: PID liveness check vs mtime-only

| Option | Description | Selected |
|--------|-------------|----------|
| Mtime-only with 90s staleness (Recommended) | Pure Node, no platform-specific code | ✓ |
| `kill -0 pid` + Windows OpenProcess equivalent | Platform-specific; faster takeover (~instant) but more code | |

**Auto-selected:** Option 1. 90s lag on crash takeover is acceptable for a presence extension. Defer platform-specific PID checks to v0.2.0+ if users complain.

---

## Logging verbosity

| Option | Description | Selected |
|--------|-------------|----------|
| Silent by default; verbose if `agentMode.debug.verbose=true` (Recommended) | Matches existing logging discipline | ✓ |
| Always log leadership transitions | Output channel noise for normal users | |
| Never log even with verbose | Hard to debug "my Discord status isn't updating" | |

**Auto-selected:** Option 1. Aligns with existing logging discipline; useful for debug.
