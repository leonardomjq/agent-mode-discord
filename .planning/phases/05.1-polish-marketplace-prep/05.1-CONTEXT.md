<decisions>
## Implementation Decisions

### Bundling
- **D-01:** Remove the `!node_modules/@xhayper/**` line from `.vscodeignore`. esbuild already bundles `@xhayper/discord-rpc` into `dist/extension.cjs` (`bundle: true`, `external: ["vscode"]` in `esbuild.mjs:42-46`). The `node_modules` re-include is dead weight that bloats the VSIX without runtime effect.
- **D-02:** Emit `dist/THIRD_PARTY_LICENSES.md` as a build-time step in `esbuild.mjs`. Walk the existing `dist/metafile.json` (already emitted in `esbuild.mjs:73`), enumerate every input from `node_modules/`, read each package's `LICENSE` file (or `license` field in `package.json` as fallback), aggregate into a single markdown file with sections per dependency. Commit the build script change; the emitted artifact lives in `dist/` (already excluded from git via `.gitignore`).

### Manifest metadata
- **D-03:** `displayName` rewrite to `"Agent Mode — Discord Presence for Claude Code & AI Agents"`. Marketplace search weights displayName heavily; "Agent Mode" alone returns nothing for the relevant searches.
- **D-04:** `categories: ["Visualization", "Other"]`. "Visualization" is the closest standard category for status/presence indicators; "Other" stays as a fallback.
- **D-05:** `keywords` array: `["discord", "rich presence", "presence", "claude", "claude code", "cursor", "ai agent", "rpc", "status"]`. These are the search terms a developer would actually type.
- **D-06:** `version` bump `0.0.1` → `0.1.0`. The whole project narrative is built around v0.1.0; the manifest should match.
- **D-07:** `icon` field pointing at `assets/icon.png`. If a 128×128 PNG of the goblin doesn't exist, this plan creates a `[HUMAN]` task entry asking the user to drop one in (the goblin pack copy is the brand — the icon is a single illustration of that brand). Marketplace listings without an icon get a generic placeholder that kills click-through.

### README polish
- **D-08:** Add a "Bus factor / using your own Client ID" subsection under the existing Privacy FAQ. Three short paragraphs: (a) the bundled Client ID is owned by the maintainer, (b) all installs ping the same Discord application, (c) if the maintainer's account is ever lost or deactivated, users can spin up their own Discord app in 2 minutes and set `agentMode.clientId` to override.
- **D-09:** Add an "Observability" section to the README explaining that the Discord Developer Portal exposes DAU/MAU/activity counts for the bundled Client ID, and that the Marketplace + OpenVSX dashboards expose install counts. Sets reader expectations and signals that the project's success is being measured (without requiring extension-side telemetry).
- **D-10:** Privacy FAQ wording fix: clarify "zero outbound HTTP" means the extension makes no HTTP requests to any third-party server. Discord IPC (Unix socket / named pipe) transmits the activity payload to the local Discord client, which forwards to Discord's servers — that path is intentional and controlled by Discord, not by us. Pedantic readers will conflate these without the clarification.

### CHANGELOG
- **D-11:** `CHANGELOG.md` follows the [Keep a Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/) format. v0.1.0 entry summarizes Phases 1–5 + Phase 5.1 in user-facing language: "Initial release. Detects Claude Code (and other AI agents) in your terminal and flips Discord status to AGENT_ACTIVE." Sections: Added / Changed / Removed / Fixed / Security. The "What's New" surface on the Marketplace listing reads from this file.

### SECURITY.md
- **D-12:** Rewrite the "Threat Model Notes" section to distinguish (a) extension network surface = zero outbound HTTP (still true), (b) Discord IPC = local-only socket / named pipe (still true), (c) what Discord's RPC protocol does after the IPC handoff = transmits the activity payload to Discord's servers via the local Discord client (their problem, not ours). Same facts, much less misleading.

### Multi-window VS Code
- **D-13:** Manual smoke test only — no automated harness in v0.1.0. Test matrix: 1 window (baseline), 2 windows same workspace, 2 windows different workspaces, 3+ windows. Document the observed behavior in `docs/MULTI-WINDOW.md`. The pid-aware `setActivity(payload, pid)` in `src/rpc/client.ts:92` should isolate each window's activity, but the *visible* behavior in Discord depends on which client connected most recently. Document, don't fix unless something is actually broken.
- **D-14:** Cursor compat: documented statement only — no live CI install test. Cursor's bundled VS Code baseline is on a known major version; document that statement in `docs/CURSOR-COMPAT.md` along with the troubleshooting steps already in the README. Live Cursor testing is captured separately as backlog `999.1`.

### IN-* findings batch
- **D-15:** Address all 7 IN-* findings from `05-REVIEW.md` in a single plan, one commit per finding. They're cosmetic (CC duplicated preposition, mtimeMs comment, version-string consistency, install command alignment, plugin.json metadata, ext install slug, CI action SHA-pinning) and grouping them avoids 7 separate plan-discussion overheads.

### Claude's Discretion
